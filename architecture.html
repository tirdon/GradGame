GradGame

A static web page that lexes calculus-style expressions, differentiates and simplifies them symbolically, and renders the result as KaTeX and as evaluable JavaScript — with a Swift core compiled to WebAssembly. The same engine drives a multiplayer artillery game on a function graph.

01System overview

GradGame is served entirely as static files — there is no backend application server. The interesting logic (parsing, differentiation, algebraic simplification, two code generators, and a deterministic game simulation) lives in a Swift package under .src/ that is compiled to a single GradGame.wasm module. The browser loads that module and talks to it across a small, hand-written C-ABI memory protocol. Everything at the repo root is the served frontend.

Browser (JS)
DOM, KaTeX, WebGPU canvas, Firebase, event wiring
FFI boundary
encode → call → decode → free over linear memory
Swift core (wasm)
lexer · parser · differentiator · simplifier · renderers · game engine

The boundary is deliberately tiny and C-compatible: all exported entry points take and return Int32 / pointers, never Swift types. Strings cross as (pointer, length) pairs in the module's linear memory.

03The core algorithm — a compilation pipeline

Turning a typed string like dx sin(x^2) into rendered math and runnable code is a textbook five-stage compiler pipeline. Each stage is a pure transformation over a shared tree (the AST); every stage recurses over that tree, which is why a hard size cap exists (see Stage 2).

InputString
e.g. dx sin(x^2) — raw UTF-8 text from the expression box.
1 · LexerExpression+Lexer.swift
bytes → [Token]. Scans UTF-8 bytes into numbers, identifiers, operators, parens, commas.
2 · ParserExpression+Parser.swift
[Token] → Expression. Recursive descent with operator precedence; builds the AST; enforces the 256-token cap.
3 · DifferentiatorOperator+Differentiator.swift
Replaces every dx/dy node with its symbolic derivative via the sum/product/quotient/power/chain rules, then canonicalizes.
4 · SimplifierOperator+Simplifier.swiftoptional
CAS-style canonicalization: constant folding, like-term collection, common-factor factoring, exact rational coefficients.
5 · RenderersRenderer+TeX.swift · Renderer+JavaScript.swift
One AST → two outputs: a KaTeX string for display and an evaluable JavaScript string for the live numeric result.
OutputTeX & JS
2 x \cos\left(x^{2}\right)  ·  2 * x * Math.cos(Math.pow(x, 2))

The same parse-and-resolve front half also feeds the game engine: parseAndResolveExpression produces an AST that evaluate(x:y:) walks numerically during the trajectory sweep, so the curve a player fires is the curve the renderer drew.

04Data structures — the AST

Every stage reads and writes one shared type: an indirect enum (a heap-boxed, recursive sum type). Numbers are kept as strings, never parsed to a fixed-width integer, so values far beyond Int32.max survive untouched until a renderer formats them. 

indirect enum Expression: Equatable {
    case number(String)                 // "2", "3.14", "100000000000"
    case variable(String)               // "x" or "y"
    case constant(String)               // "pi", "e", "phi", "gamma"
    case unary(UnaryOperator, Expression)                // +x, -x
    case binary(BinaryOperator, Expression, Expression)  // a + b, a^b …
    case function(name: String, argument: Expression,
                  exponent: Expression?, parenthesized: Bool)  // sin^2(x)
    case derivative(variable: String, argument: Expression,
                    parenthesized: Bool)   // dx(...) — resolved away by stage 3
}

enum BinaryOperator { case add, subtract, multiply, implicitMultiply, divide, power }
Why a separate implicitMultiply

Juxtaposition (2 x, x sin(x)) parses to .implicitMultiply, distinct from an explicit *. The renderers exploit the difference: TeX prints implicit products with a thin space (2 x) and explicit ones with \times.

Supporting numeric types live alongside: DecimalValue (a mantissa × 10exp form used for overflow-safe folding) and Rational (an exact numerator/denominator coefficient so the simplifier can fold 2/3 · 4/5 → 8/15).

05Stage 1 — Lexer

The lexer walks the input as a [UInt8] byte array (UTF-8) with a single index, emitting a flat token list terminated by .end.

Procedure

  • Skip whitespace (space, tab, CR, LF).
  • Single-byte punctuation → its token: + - * / ^ , ( ).
  • A digit or . starts a number: consume digits and at most one decimal point. At least one digit must appear, else error.
  • A letter [A–Za–z] starts an identifier: consume consecutive letters.
  • Anything else → unexpectedCharacter.
No E-notation

2E3 does not lex as a single number — it becomes 2, identifier E, 3, which the parser rejects. Write a power of ten explicitly: 2 10^3. (Lowercase e is Euler's constant.)

Tokens: number(String), identifier(String), plus, minus, star, slash, caret, comma, leftParenthesis, rightParenthesis, end.

06Stage 2 — Parser

A classic recursive-descent parser, one function per precedence level. Lower-precedence rules call higher ones, so binding strength falls out of the call structure. The grammar (lowest to highest precedence): 

parseAddition       → parseMultiplication ( ('+' | '-') parseMultiplication )*      // left-assoc
parseMultiplication → parseUnary ( ('*' | '/' | implicit) parseUnary )*            // left-assoc
parseUnary          → ('+' | '-') parseUnary | parsePower
parsePower          → parsePrimary ( '^' parseUnary )?                            // right-assoc exponent
parsePrimary        → number | identifier-form | '(' parseAddition ')'
FormRecognized as
x, yvariable
pi, e, phi, gammaconstant
sin cos tan sec csc cot sinh cosh tanh exp log sqrtunary function — supports sin^2 x and bare-argument sin x
pow(a, b)desugars to .binary(.power, a, b)
dx, dyderivative operator — dx(f) or bare dx f
juxtaposition 2 x, x(y+1).implicitMultiply
Recursion depth = expression size

Parsing, simplification, and both renderers all recurse over the tree, and the wasm stack is small. The parser caps input at maximumTokenCount = 256 and throws expressionTooComplex rather than overflowing the stack. Bounding tokens bounds every later stage's recursion depth — this guard is load-bearing.

07Stage 3 — Differentiator

resolve() walks the tree and replaces every derivative node with the actual derivative of its argument, then hands that raw result to the simplifier. The rules are the standard calculus identities — themselves the core algebraic properties applied to d/dv:

Rules applied

  • Linearity: d(f ± g) = df ± dg
  • Product: d(f·g) = df·g + f·dg
  • Quotient: d(f/g) = (df·g − f·dg) / g²
  • Power: d(fⁿ) = n·fⁿ⁻¹·df
  • Exponential: d(eᵘ) = eᵘ·du
  • Chain: d(F(u)) = F'(u)·du

Partial derivatives

dx holds y (and every symbolic constant e, pi, phi, gamma) constant — their derivative is 0; dy holds x constant likewise. A built-in dependsOn() traversal decides whether a sub-tree varies in the differentiation variable, which selects the power vs. exponential vs. symbolic branch.

The one case left symbolic

A derivative that genuinely needs a logarithm — a non-e base with a variable exponent (2^x) or a variable base and exponent (x^x) — is deliberately not resolved. It stays a derivative node: the TeX renderer shows ∂-notation, and the JS renderer falls back to a numeric central difference.

08Stage 4 — Simplifier

A deliberately scoped computer-algebra system (run unconditionally after differentiation, and again when the user asks for a simplified display). It applies the three core algebraic properties:

PropertyWhat it does
CommutativeCanonical ordering of terms/factors via parallel [String] sort keys + a manual insertion sort.
AssociativeFlattens nested +/· into n-ary groups, then collects like terms and merges repeated bases into powers.
DistributiveFactors out a common factor (the numeric GCD of coefficients plus shared symbolic bases at their minimum exponent).

Numeric folding

  • Integer constants fold only within float32 range (decimal exponent ≤ 38); a larger magnitude stays symbolic, and a magnitude below 10-39 underflows to exact zero.
  • Overflow is detected with a Double-magnitude check plus wrapping &*/&+ — never the stdlib reporting-overflow APIs (they trap on this wasm SDK).
  • Coefficients are exact rationals (Rational over DecimalValue): 4x/2 → 2x, 6/4 → 3/2.
Division by zero throws

c / 0 for a non-zero numerator throws divisionByZero; 0 / 0 throws notANumber. (Earlier this folded to an internal infinity constant, but a later 0 · e → 0 rule could swallow it, so 1 / 0 · 0 wrongly became 0.) Surfacing these as thrown errors keeps the simplifier from carrying a value downstream rules must each special-case.

09Stage 5 — Renderers

Two renderers walk the same AST and emit two strings. Both decide parenthesization from a single shared module, Renderer+Precedence.swift, so the TeX and JS outputs always agree on grouping — only the delimiters and spellings differ.

TeXRenderer → KaTeX

  • Builds macros: \frac{}{}, \sqrt{}, \times, \cos, ∂-notation.
  • Implicit products print as a thin space; explicit as \times.
  • renderNumber formats huge values as mantissa \times 10^{exp} by scanning digit bytes — no overflow.

JavaScriptRenderer → eval string

  • Emits Math.sin, Math.pow, Math.PI, etc.
  • The page evals it with the current x, y for a live numeric result.
  • For the symbolic-derivative fallback it emits a small central-difference closure.

Shared precedence & parenthesization

atoms / functions ............ 5
power  (^) ................... 4

multiply / implicit / divide . 2

10The game engine — "Graph War"

The front half of the pipeline (parse + resolve) also powers an artillery game: players are cannons on a battlefield, and a turn fires the player's expression as a curve. The engine is pure Swift math over the AST (no DOM/Firebase), unit-testable on the host and exposed to JS through its own FFI exports.

Trajectory simulation (simulateShot)

// Normalize so the arc passes through the firing cannon:
offset = expression.evaluate(x: 0, y: 0)        // nil-out the shot if f(0) is non-finite
f(s)   = expression.evaluate(x: s, y: 0) − offset

for each step (x advances by dir · kStep=0.03 from the cannon):
    wy = originY + f(x − originX)
    if x left the field .......................→ outcome = OUT
    if within kCannonRadius(0.55) of a live enemy → outcome = HIT  (record seat)
    if within an obstacle radius ...............→ outcome = BLOCKED
    // collisions ignored within kMuzzleClearance (0.7) of the firer
    record (round3(x), round3(wy)) into the flat path [x0,y0,x1,y1,…]

round3 rounds exactly like JS Math.round (floor(v·1000 + 0.5)/1000) so paths match the original JavaScript engine bit-for-bit. resampleArc rebuilds just the visible polyline (no collision tests) up to the recorded impact, so every client redraws the same arc regardless of since-changed state — the shot itself stores no path.

Deterministic setup

Board layout is reproducible from a seed via a 31-bit LCG (glibc constants, wrapping &*/&+ masked to 0x7FFFFFFF) so it behaves identically on the host (64-bit Int) and wasm (32-bit Int): placePlayers spaces cannons ≥ 5 units apart, and generateObstacles drops 3–6 circles avoiding cannons. Seats and turn order are tracked as bitmasks (aliveCount, nextAliveSeat).

11Procedure — the wasm FFI boundary

The browser cannot hand a Swift String to wasm, so there is a manual memory protocol. Exported functions are marked @_expose(wasm, …) + @_cdecl and only speak Int32 / pointers.

The string call dance (parser exports)

// JS side, for parseExpressionToTex / parseExpressionToJavaScript:
1. ptr = gradGameAllocate(utf8.length)          // reserve linear memory
2. copy the UTF-8 bytes into memory at ptr
3. resultPtr = parseExpressionToTex(ptr, len, simplify)
4. len  = gradGameLastResultLength()           // result stashed in module globals
5. ok   = gradGameLastParseSucceeded()         // 1 = parsed, 0 = error string
6. decode UTF-8 at (resultPtr, len)
7. gradGameFreeLastResult()  and  gradGameDeallocate(ptr, len)

On any parse error the export still returns a string — the human-readable ExpressionParserError.description — with the "succeeded" flag set to 0, so a bad input is shown as a message and never traps the instance.

Exported entry points

ExportPurpose

Array results — host callback, not a buffer

Array results (trajectory paths, placements, obstacles) are pushed back through a host import gradgame.gradGameEmitResult(outcome, hitSeat, impactX, impactY, pathPtr, pathCount) that JS supplies at instantiation. The path pointer is valid only during the call, so JS copies it out synchronously. Omit this import and WebAssembly.instantiate raises a LinkError.

12Procedure — frontend glue

  • Boot: gradgame-wasm.js fetches and instantiates GradGame.wasm with a wasi_snapshot_preview1 stub and the gradgame import module, runs the add(2,3) smoke test, then resolves window.gradGameReady and installs window.gradGameEngine / gradGameParse / gradGameConfig.
  • Live render: input is debounced ~120 ms (scheduleRender); each render does the call dance, hands the TeX to KaTeX, and evals the JS form to show f(x,y).
  • Trap recovery: a trapped call throws a WasmTrapError and triggers recoverWasm(), which rebuilds a fresh instance from the cached Module. This is a backstop — the real fix for any freeze is preventing the Swift-side trap.
  • Game loop: gradgame-graph.js draws the grapher/battlefield on a WebGPU canvas; gradgame-game.js drives turns and syncs state over Firebase RTDB. Writes are client-authoritative (a client may write another player's seat), and resampleArc lets every client reproduce a shot's arc without storing the path.

13wasm constraints (why the Swift looks the way it does)

On wasm32, Swift's Int is 32-bit; on the host it is 64-bit. More importantly, the swift-6.3.2-RELEASE_wasm SDK fails to resolve certain runtime metadata, so code that passes host tests can still hard-crash the wasm instance (an unreachable trap that freezes the input box). These patterns are avoided throughout .src:

Avoid (traps on wasm)Use instead
Int32/Int64/UInt string interpolation, .magnitude interpolationInt bounded to ±Int32.max so host and wasm agree
for x in someString.utf8 and other Sequence for-inlet bytes = Array(text.utf8) + a while index loop
Generic Array map/sorted(by:)/flatMap/firstIndex(where:)/+Manual index loops & insertion sort with parallel [String] keys
Stdlib reporting-overflow (multipliedReportingOverflow, …)Double-magnitude check + wrapping &*/&+

Host unit tests are fast but cannot reproduce wasm-only traps, so behavior is always verified against the actual GradGame.wasm via a Node harness in addition to swift test.

14Build & run procedure

# Host unit tests (Swift Testing). Fast, but does NOT catch wasm-only traps.
swift test --package-path .src --scratch-path .build

# Build the wasm artifact, then copy it to where index.html fetches it.
swift build --package-path .src --scratch-path .build --swift-sdk swift-6.3.2-RELEASE_wasm
cp .build/wasm32-unknown-wasip1/debug/GradGame.wasm GradGame.wasm

# Serve from the repo root (the page fetches GradGame.wasm — file:// will not work).
bunx serve . -l 8000

After rebuilding GradGame.wasm, hard-reload the page (⌘⇧R) to bypass the browser cache.
